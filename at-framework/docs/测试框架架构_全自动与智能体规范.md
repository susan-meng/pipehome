# 测试框架架构：全自动化 + 智能体规范

本架构以**全流程自动化、零人工参与**为前提，以**规范（Spec）为核心**，使智能体能够按规范完成用例的增删改与按需提取、执行。

---

## 一、设计原则

1. **全自动化**：从“提交监听 → 范围决策 → 用例增删改 → 用例提取 → 执行”全程由流水线/智能体完成，无需人工介入。
2. **架构即契约**：用例结构、套件结构、全局变量、API 定义均有唯一规范（Spec），智能体仅需遵循规范即可正确增删改与提取。
3. **全局变量可提取**：全局变量集中定义、可被显式列举（Manifest），智能体可查询“有哪些变量、如何引用”，避免硬编码与遗漏。

---

## 二、整体自动化流水线

```
[研发仓库 push] 
    → 提交监听服务（变更路径 + 类型 + 受影响 API/文件）
    → 路径→scope 映射 + 受影响 API 列表（api_name / api_path）
    → 若需补用例：智能体读取 Spec + 建议清单 → 自动增/删/改 suites/*.yaml
    → 按「单条用例」筛选用例：get_cases(api_name=..., api_path=..., tags=...) 得到本次要跑的用例列表（一个个筛，不是整 suite）
    → 执行器只跑上述用例（pytest 仅 parametrize 这些 case）
    → 报告产出（Allure/JUnit）
```

**重要**：筛选粒度是**单条用例**。有新的提交时，根据实际变更（如改动了哪个 API、哪些路径）只筛出并运行**受影响的用例**，而不是整个 suite 或整批 tag 一起跑。

---

## 三、目录与契约布局（多模块通用）

框架**通用所有测试模块**，VEGA 仅为其中一个模块；目录分为**框架级**与**模块级**。

```
testcase/
  _config/                      # 框架级（所有模块共用）
    spec/
      case_schema.yaml           # 单条 case 字段规范（智能体增删改依据）
      suite_schema.yaml         # 套件顶层规范（feature, story, switch, tags, cases）
    README.md

  vega/                         # 模块：VEGA（可按同样结构新增其他模块）
    _config/
      global.yaml               # 本模块全局变量
      global_manifest.yaml      # 本模块变量清单（供智能体提取）
      apis.yaml                 # 本模块接口定义
      path_scope_mapping.yaml   # 本模块：提交路径 → scope/tags/建议套件
      suite_manifest.yaml       # 本模块套件列表说明
      spec/README.md            # 指向框架级 spec，勿在此放 schema
    suites/
      *.yaml                    # 本模块各套件，符合框架 spec

  其他模块/                     # 未来其他产品线同理：<模块名>/_config + suites
```

- **运行哪个模块**：`config.ini` 中 `case_file = ./testcase/vega`（或 `./testcase/其他模块`）；执行/提取时以该目录为 `base_dir`。
- **智能体增删改 case**：只读**框架** `testcase/_config/spec/case_schema.yaml`、`suite_schema.yaml` + **本模块** `apis.yaml`、`global_manifest.yaml`，按规范编辑该模块下 `suites/*.yaml`。
- **按需提取 case**：对某模块目录调用 `get_cases(base_dir, scope=..., tags=..., api_name=..., api_path=...)`，**粒度到单条用例**。
- **执行**：通过环境变量 `SCOPE`/`TAGS`/`API_NAME`/`API_PATH` 等筛选后，框架只跑当前 `case_file` 对应模块的匹配用例。

## 四、全局变量：定义、提取与使用

### 4.1 唯一来源与解析顺序

- **唯一来源**：各模块目录下 `_config/global.yaml`。  
  - 格式：`name: value` 或 `[{name, value}, ...]`。  
  - 支持变量间引用：先做一轮 `string.Template` 替换（如 `${mysql_host}`），得到 `global_flat`，再参与用例参数替换。

- **在用例中的使用顺序**（执行时）：  
  1. **global_flat**（来自 global.yaml）  
  2. **resp_values**（前序用例 jsonpath 提取）  
  3. **path_params**（当前用例路径参数）  
  4. 对字符串再做 **Jinja2** 渲染（若需要）。  

  替换语法：用例中写 `$var` 或 `${var}`，由框架用 `global_flat` 与 `resp_values` 做 `string.Template.safe_substitute`。

### 4.2 全局变量清单（各模块 _config/global_manifest.yaml）— 供智能体提取

目的：智能体无需解析业务含义，即可知道该模块“有哪些全局变量、如何引用、用于何种场景”。

```yaml
# 说明：列出所有可在 case 中引用的全局变量；由流水线或人工与 global.yaml 同步
variables:
  - name: mysql_host
    description: MySQL 主机地址
    ref: "${mysql_host}"
  - name: mysql_port
    description: MySQL 端口
    ref: "${mysql_port}"
  - name: bin_data_template_mysql
    description: MySQL 数据源连接 JSON 模板，用于 body_params
    ref: "${bin_data_template_mysql}"
    used_in: [body_params]
  # ... 其他变量
```

- **提取接口**：框架提供 `load_global_manifest(base_dir)`，对某模块目录返回该模块变量名列表或完整 manifest；智能体据此生成/校验 case 中的 `$var` 引用。
- **与 global.yaml 的同步**：可由脚本根据该模块 `global.yaml` 的 key 自动生成 manifest 骨架（description 可后续补全），保证不遗漏。

### 4.3 实现要点（代码侧）

- `load_global_manifest(base_dir)`：读取该模块 `_config/global_manifest.yaml`，返回 `List[Dict]`。
- `get_global_flat(base_dir)`：读该模块 `_config/global.yaml` → 解析为 `global_flat`（用于替换）。
- 文档约定：case 中只使用本模块 `global.yaml` 中已定义的 key 或 manifest 中列出的 name，避免运行时未定义引用。

## 五、用例与套件规范（Spec）— 智能体增删改依据

规范为**框架级**，位于 `testcase/_config/spec/`，**所有模块共用**；智能体对任意模块增删改 case 时均以该 spec 为准。

### 5.1 单条 case 规范（case_schema.yaml）

智能体**新增/修改 case** 时，必须且仅能使用以下字段，类型与含义固定：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 用例唯一名，建议「场景_预期」 |
| url | string | 是 | 接口名称，与本模块 apis.yaml 的 name 一致 |
| prev_case | string | 否 | 前置用例 name，空表示无 |
| path_params | string(JSON) | 否 | 路径参数对象，如 `{"id":"${catalog_id}"}` |
| query_params | string(JSON) | 否 | 查询参数 |
| body_params | string(JSON) | 否 | 请求体；可含 `${var}`、`${bin_data_template_xxx}` |
| form_params | string(JSON) | 否 | 表单 |
| resp_values | string(JSON) | 否 | jsonpath 提取到后续用例，如 `{"catalog_id":"$.id"}` |
| code_check | string | 否 | 期望 HTTP 状态码 |
| resp_check | string(JSON) | 否 | jsonpath 期望，如 `{"$.code":0}` |
| resp_schema | string(JSON) | 否 | 响应样本，用于 genson 生成 schema 校验 |
| **tags** | **list[string]** | **否** | **本条用例的标签（粒度到单条）；未写则继承套件 tags。用于按提交精确筛选用例，如 [api:测试数据源连接], [smoke]** |

引用全局变量时使用 `$var` 或 `${var}`，执行时由框架用 global_flat + resp_values 替换。

**tags 粒度**：tags 落在**单条用例**上。有新提交时，按 `api_name`/`api_path` 或 case 的 tags 一个个筛选用例，只跑受影响的用例，而不是整 suite 运行。

### 5.2 套件规范（suite_schema.yaml）

每个 `suites/*.yaml` 文件结构：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| feature | string | 是 | 功能大类 |
| story | string | 是 | 场景/故事 |
| switch | string | 是 | y/n，n 则不加载 |
| tags | list[string] | 否 | 套件默认标签；单条 case 未写 tags 时继承此处；若 case 自带 tags 则以 case 为准 |
| cases | list[case] | 是 | 符合 case_schema 的数组 |

智能体**新增套件**或**修改 suite tags** 时按此结构；**删除 case** 即从 `cases` 中移除对应项；**修改 case** 即替换同 name 的项；单条 case 可带自己的 **tags** 以实现按提交逐条筛选。

## 六、按需提取用例（get_cases）— 粒度到单条

框架提供**按单条用例筛选**的提取，有新的提交时按实际变更（受影响 API、路径或 case tags）一个个筛选用例，**不整 suite 运行**。

- **接口形态**：`get_cases(base_dir, scope=None, tags=None, suite=None, name=None, names=None, api_name=None, api_path=None)`  
  - **`names`**：**用例 name 列表**；只保留 name 在此列表中的用例。智能体根据提交内容与 case 的 name/description 灵活选出要回归的 case 名后传此参数，**不依赖固定 tag**。  
  - `scope`：与 path_scope_mapping 对齐，转为 tags，再按**单条 case 的 tags** 过滤。  
  - `tags`：列表，**单条 case 的 tags** 与其中任一相交则保留。  
  - `suite`：story 或文件名，精确到某套件。  
  - `name`：用例 name 精确匹配（单条）。  
  - **`api_name`**：接口名称（与 apis.yaml 的 name 一致），只保留**调用该接口的用例**（一条条筛）。  
  - **`api_path`**：接口路径（如 `/api/data-connection/v1/datasource/test`），只保留 url 与该路径匹配的用例。  
  - 返回：与 load_case_from_yaml 同结构的 case 列表（含 `api_name`、tags、_suite_file），便于直接 parametrize 或导出。

- **与执行的关系**：提交监听输出「受影响的 api_name / api_path 或 case 列表」后，执行前调用 `get_cases(..., api_name=..., api_path=..., tags=...)` 得到**本次要跑的单条用例**，再交给 pytest；实现“按提交一个个筛选用例”。

---

## 七、提交监听与范围决策（无人工）

- **输入**：commit 变更文件列表、变更类型（增/改/删）、可选 diff（如受影响 API 路径）。  
- **规则表**：各模块 `_config/path_scope_mapping.yaml`（如 vega 的路径对应 vega 的 scope/tags）；并可根据代码/OpenAPI 解析出**受影响的 API**（api_name 或 api_path）。  
- **逻辑**：  
  1. 对每个变更路径匹配路径模式，得到 scope、tags、suggested_suites。  
  2. 解析本次提交**实际影响的 API**（如改动了 `data-connection` 的 test 接口 → api_name=测试数据源连接 或 api_path=/api/data-connection/v1/datasource/test）。  
  3. 输出：`scope`、`tags`、**`affected_api_names`**、**`affected_api_paths`**、`suggested_suites`、`need_add_cases`、可选的「建议变更清单」。  
- **下游**：  
  - **执行**：用 `get_cases(..., api_name=..., api_path=..., tags=...)` 按**单条用例**筛出本次要跑的 case（只跑受影响的用例，不整 suite），再执行。  
  - 补用例：若 `need_add_cases` 为 true，智能体在 suggested_suites 中增/删/改 case，并为新 case 打上细粒度 tags（如 api:xxx）便于后续按提交筛选。

## 八、智能体自动化动作清单（无人工）

| 动作 | 输入 | 规范依据 | 输出 |
|------|------|----------|------|
| 新增 case | 建议清单 + API 名 + 场景 | 框架 case_schema + 本模块 apis.yaml, global_manifest | 追加到该模块 suites/xxx.yaml 的 cases |
| 修改 case | 用例 name + 新字段 | case_schema | 替换同 name 的 case |
| 删除 case | 用例 name 或 API 下线 | suite_schema | 从 cases 中移除或 suite switch=n |
| 提取 case | scope / tags / suite / name / **names** / api_name / api_path | get_cases API（粒度到单条）；**names** 为智能体根据提交内容选出的 case 名列表 | JSON/YAML case 列表 |
| 执行 | **CASE_NAMES**（逗号分隔的 case name）/ scope / tags / API_NAME / API_PATH | get_cases 筛出单条用例 + pytest；**CASE_NAMES 优先**，按 name 灵活回归 | 只跑筛出的用例，不整 suite |

全局变量相关：智能体**只读 global_manifest** 获取可用变量名与引用方式，**不**在 case 中写未在 manifest 或 global.yaml 中出现的变量。

**测试类型（功能 / 契约 / 性能）**  
除 API 功能测试、契约测试外，涉及性能的改动需能识别并执行性能用例。约定如下：  
- **功能**：常规 API 行为与回归，tags 如 `smoke`、`regression`。  
- **契约**：结构/字段校验（如 `resp_schema`），通常与 regression 共用 tags。  
- **性能**：用例或套件打 tag **`performance`**；智能体通过 **`TAGS=performance`** 或 **`SCOPE=performance`** 筛选并执行。  
- `path_scope_mapping.yaml` 中 `test_type_tags.performance: [performance]`，子系统可配置 `performance_tags`；提交监听若判定为“性能相关改动”，可输出建议执行 `TAGS=performance` 或 `SCOPE=performance`。

## 九、智能体规范速查（零人工）

| 动作 | 输入 | 规范/入口 | 说明 |
|------|------|-----------|------|
| 查全局变量 | 某模块 base_dir | 该模块 `_config/global_manifest.yaml` 或 `CASE_FILE=testcase/vega python scripts/extract_cases.py --globals` | 该模块变量名、ref、description |
| 按单条筛选用例 | **names**（case 名列表）/ scope / tags / suite / name / api_name / api_path | `get_cases(..., names=[...])` 或 `extract_cases.py --names "名1,名2"`；先 `--list-fields name,description,story` 拉全量供智能体按提交内容选 name | 根据提交内容灵活选 name，不依赖固定 tag |
| 按单条执行 | 环境变量 **CASE_NAMES**（逗号分隔）/ SCOPE / TAGS / API_NAME / API_PATH | **`CASE_NAMES=用例名1,用例名2 pytest`**（智能体选出 name 后）；或 TAGS=smoke / SCOPE=performance 等 | CASE_NAMES 优先，按 name 执行；否则按 scope/tags/api 执行 |
| 新增/修改/删除 case | 建议清单 + 框架 case_schema | 编辑**该模块** `suites/*.yaml`，符合 `testcase/_config/spec/case_schema.yaml`、`suite_schema.yaml` | url 用本模块 apis.yaml 的 name；引用变量用本模块 global_manifest 中的 ref |

## 十、小结

- **全自动化**：提交 → 范围 + 受影响 API → 补用例（智能体）→ **按单条筛选用例** → 执行，全链路无人工。  
- **粒度到单条用例**：tags 落在单条 case 上；筛选支持 api_name、api_path、tags 等，有提交时按实际**一个个筛选用例**，不整 suite 运行。  
- **核心是架构/规范**：**框架级** case_schema、suite_schema（`testcase/_config/spec/`）与**各模块** global_manifest、apis、path_scope_mapping 构成契约；智能体按规范对指定模块增删改与提取。
- **多模块**：VEGA 为其中之一；切换模块只需修改 `case_file` 指向不同 `testcase/<模块名>`。

**如何新增模块**：在 `testcase/` 下新建目录 `<模块名>`，其下创建 `_config/`（apis.yaml、global.yaml、global_manifest.yaml、path_scope_mapping.yaml、suite_manifest.yaml）与 `suites/`（*.yaml）；规范统一使用 `testcase/_config/spec/`。在 config.ini 中将 `case_file` 改为 `./testcase/<模块名>` 即可运行该模块。  
- **全局变量**：单一来源 global.yaml，manifest 显式列举可引用变量及用法；替换顺序为 global_flat → resp_values → path_params → Jinja2。
