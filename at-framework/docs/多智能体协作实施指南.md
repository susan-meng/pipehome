# 多智能体协作实施指南

本文档为**多智能体协作部分**的实施指南，帮助你理解如何实现和集成四个智能体（A/B/C/D）到测试流水线中。

---

## 一、整体架构理解

### 1.1 智能体协作流程

```
提交监听 → 智能体 A（用例维护） + 智能体 B（维度分析）并行
    ↓
智能体 C（用例筛选与执行）
    ↓
执行器（pytest）
    ↓
智能体 D（报告生成）
```

### 1.2 关键原则

- **CI/Workflow 编排**：由 GitHub Actions 等 CI 工具编排，规则驱动，无需主控 Agent
- **标准化输入输出**：每个智能体有明确的输入输出格式（JSON）
- **规范即契约**：所有智能体必须遵循 `case_schema.yaml`、`suite_schema.yaml` 等规范
- **单条用例粒度**：筛选和执行都精确到单条用例，不整 suite 执行

---

## 二、智能体 A：用例维护智能体（Case Maintenance Agent）

### 2.1 职责

根据 commit 变更，对现有 YAML 用例进行**增、删、改**操作。

### 2.2 输入

从 CI 接收 `listener_output.json`（提交监听服务的输出）：

```json
{
  "repo": "owner/repo",
  "branch": "main",
  "commit_sha": "abc123...",
  "commit_message": "feat: add datasource test API",
  "changed_files": [...],
  "scopes": ["vega-data-connection"],
  "suggested_suites": ["测试数据源", "新增数据源"],
  "affected_api_names": ["测试数据源连接", "新增数据源"],
  "affected_api_paths": ["/api/data-connection/v1/datasource/test"],
  "need_add_cases": true,
  "diffs": [...]
}
```

**还需要读取**：
- `testcase/_config/spec/case_schema.yaml`（用例规范）
- `testcase/_config/spec/suite_schema.yaml`（套件规范）
- `testcase/vega/_config/apis.yaml`（接口定义）
- `testcase/vega/_config/global_manifest.yaml`（全局变量清单）
- `testcase/vega/suites/*.yaml`（现有用例）

### 2.3 实现步骤

#### 步骤 1：判断是否需要执行

```python
if not listener_output.get("need_add_cases", False):
    # 快速退出，不修改用例
    return {"status": "skipped", "reason": "no need to add cases"}
```

#### 步骤 2：分析变更内容

```python
# 解析提交信息
commit_message = listener_output["commit_message"]
affected_apis = listener_output["affected_api_names"]
suggested_suites = listener_output["suggested_suites"]
diffs = listener_output.get("diffs", [])

# 判断变更类型
# - 新增 API → 需要新增用例
# - 修改 API → 可能需要修改现有用例或新增用例
# - 删除 API → 需要删除或下线用例
```

#### 步骤 3：读取规范文件

```python
import yaml

# 读取规范
with open("testcase/_config/spec/case_schema.yaml") as f:
    case_schema = yaml.safe_load(f)

with open("testcase/vega/_config/apis.yaml") as f:
    apis = yaml.safe_load(f)

with open("testcase/vega/_config/global_manifest.yaml") as f:
    global_manifest = yaml.safe_load(f)

# 读取现有用例
existing_cases = load_case_from_yaml("testcase/vega")
```

#### 步骤 4：生成/修改用例

**新增用例示例**：

```python
# 找到对应的 suite 文件
suite_file = "testcase/vega/suites/测试数据源.yaml"

# 读取现有 suite
with open(suite_file) as f:
    suite = yaml.safe_load(f)

# 生成新用例（符合 case_schema）
new_case = {
    "name": "测试数据源连接_参数校验_type缺省_请求失败",
    "url": "测试数据源连接",  # 必须与 apis.yaml 中的 name 一致
    "prev_case": "",
    "path_params": "",
    "query_params": "",
    "body_params": '{"type": "", "bin_data": ${bin_data_template_mysql}}',  # 使用 global_manifest 中的变量
    "code_check": "400",
    "resp_check": "",
    "tags": ["api:测试数据源连接", "regression"]  # 细粒度 tags
}

# 追加到 cases
suite["cases"].append(new_case)

# 写回文件
with open(suite_file, "w", encoding="utf-8") as f:
    yaml.dump(suite, f, allow_unicode=True, default_flow_style=False)
```

**修改用例**：

```python
# 找到要修改的用例（按 name）
for case in suite["cases"]:
    if case["name"] == "要修改的用例名":
        # 更新字段
        case["body_params"] = '{"type": "mysql", ...}'
        break
```

**删除用例**：

```python
# 从 cases 中移除
suite["cases"] = [c for c in suite["cases"] if c["name"] != "要删除的用例名"]
```

### 2.4 输出

**写回方式（推荐 PR）**：

```python
# 1. 创建新分支
branch_name = f"auto/{listener_output['branch']}-{listener_output['commit_sha'][:7]}"
# git checkout -b {branch_name}

# 2. 提交变更
# git add testcase/vega/suites/*.yaml
# git commit -m "[auto] 用例维护 @{commit_sha}"

# 3. 推送并创建 PR
# git push origin {branch_name}
# 使用 GitHub API 创建 PR，base 为 main
```

**输出 JSON**（供 CI 使用）：

```json
{
  "status": "success",
  "modified_files": ["testcase/vega/suites/测试数据源.yaml"],
  "pr_url": "https://github.com/owner/repo/pull/123",
  "summary": "新增 3 个用例，修改 1 个用例"
}
```

### 2.5 注意事项

- ✅ 必须遵循 `case_schema.yaml` 的所有字段规范
- ✅ `url` 字段必须与 `apis.yaml` 中的 `name` 完全一致
- ✅ 变量引用必须使用 `global_manifest.yaml` 中列出的变量
- ✅ 写回前需校验分支 tip 是否仍为当前 `commit_sha`（避免并发冲突）
- ✅ 新用例必须打上细粒度 tags（如 `api:xxx`）便于后续筛选

## 三、智能体 B：测试维度分析智能体（Test Dimension Analyzer Agent）

### 3.1 职责

分析本次 commit 需要执行哪些测试维度：功能、契约、边界、压力、性能。

### 3.2 输入

- `listener_output.json`（提交监听输出）
- 现有用例的 tags 和类型信息

### 3.3 实现步骤

#### 步骤 1：分析变更内容

```python
# 读取提交信息
commit_message = listener_output["commit_message"]
changed_files = listener_output["changed_files"]
diffs = listener_output.get("diffs", [])

# 判断变更类型
has_api_change = any("Controller" in f["path"] for f in changed_files)
has_validation_change = any("校验" in commit_message or "validation" in commit_message.lower())
has_performance_change = any("性能" in commit_message or "performance" in commit_message.lower())
has_concurrent_change = any("并发" in commit_message or "concurrent" in commit_message.lower())
```

#### 步骤 2：判断测试维度

```python
dimensions = ["functional"]  # functional 必含

# 契约测试：接口定义变更、请求/响应结构变更
if has_api_change or any("schema" in diff.get("patch", "") for diff in diffs):
    dimensions.append("contract")

# 边界测试：参数校验逻辑变更
if has_validation_change:
    dimensions.append("boundary")

# 压力测试：并发/限流/熔断相关变更
if has_concurrent_change:
    dimensions.append("stress")

# 性能测试：性能相关路径或配置
if has_performance_change or "vega-performance" in listener_output.get("scopes", []):
    dimensions.append("performance")
```

### 3.4 输出

```json
{
  "dimensions": ["functional", "contract", "boundary"],
  "reason": "接口定义变更，建议执行契约测试；参数校验逻辑变更，建议执行边界测试"
}
```

**输出格式规范**：
- `dimensions`：数组，必须包含 `functional`，其他按需
- `reason`：字符串，说明判断依据

## 四、智能体 C：用例筛选与执行智能体（Case Selector & Executor Agent）

### 4.1 职责

根据优先级策略，筛选出要执行的用例，并输出执行批次。

### 4.2 输入

- `listener_output.json`（提交监听输出）
- `agent_b_output.json`（智能体 B 的 dimensions 输出）

### 4.3 实现步骤

#### 步骤 1：按优先级筛选用例

```python
from common.func import get_cases

base_dir = "testcase/vega"
batches = []

# 第一优先级：新增功能相关
affected_api_names = listener_output["affected_api_names"]
if affected_api_names:
    cases_priority1 = get_cases(
        base_dir,
        api_name=affected_api_names[0]  # 或遍历所有 affected_api_names
    )
    batches.append({
        "priority": 1,
        "label": "新增功能",
        "case_names": ",".join([c["name"] for c in cases_priority1])
    })

# 第二优先级：回归相关
scopes = listener_output["scopes"]
if scopes:
    cases_priority2 = get_cases(
        base_dir,
        scope=scopes[0],
        tags=["regression"]
    )
    # 排除第一优先级已选的用例
    existing_names = set(batches[0]["case_names"].split(","))
    cases_priority2 = [c for c in cases_priority2 if c["name"] not in existing_names]
    
    if cases_priority2:
        batches.append({
            "priority": 2,
            "label": "回归",
            "case_names": ",".join([c["name"] for c in cases_priority2])
        })

# 第三优先级：按维度评估
dimensions = agent_b_output["dimensions"]
if "boundary" in dimensions:
    cases_boundary = get_cases(base_dir, tags=["boundary"])
    batches.append({
        "priority": 3,
        "label": "边界测试",
        "case_names": ",".join([c["name"] for c in cases_boundary])
    })
```

#### 步骤 2：处理依赖链

```python
# 重要：必须包含所有 prev_case 依赖链
def expand_dependencies(case_names, all_cases):
    """展开用例依赖链"""
    expanded = set(case_names)
    for case_name in case_names:
        case = next((c for c in all_cases if c["name"] == case_name), None)
        if case and case.get("prev_case"):
            expanded.add(case["prev_case"])
            # 递归处理前置用例的依赖
            expanded.update(expand_dependencies([case["prev_case"]], all_cases))
    return list(expanded)

# 对每个 batch 的 case_names 展开依赖
for batch in batches:
    all_cases = load_case_from_yaml(base_dir)
    expanded_names = expand_dependencies(
        batch["case_names"].split(","),
        all_cases
    )
    batch["case_names"] = ",".join(expanded_names)
```

### 4.4 输出

**方案一（按 name 执行）**：

```json
{
  "mode": "case_names",
  "batches": [
    {
      "priority": 1,
      "label": "新增功能",
      "case_names": "用例名1,用例名2,用例名3"
    },
    {
      "priority": 2,
      "label": "回归",
      "case_names": "用例名4,用例名5"
    }
  ]
}
```

**方案二（按 filter 执行）**：

```json
{
  "mode": "filter",
  "batches": [
    {
      "priority": 1,
      "label": "新增功能",
      "api_name": "测试数据源连接",
      "api_path": null,
      "scope": null,
      "tags": null
    }
  ]
}
```

## 五、智能体 D：报告生成智能体（Report Generator Agent）

### 5.1 职责

汇总测试结果，生成可读的测试报告。

### 5.2 输入

- `report/junit_report.xml`（JUnit 测试结果）
- `report/xml/`（Allure XML 结果）
- 执行上下文（dimensions、batches 等）

### 5.3 实现步骤

#### 步骤 1：解析测试结果

```python
import xml.etree.ElementTree as ET

# 解析 JUnit XML
tree = ET.parse("report/junit_report.xml")
root = tree.getroot()

total = int(root.get("tests", 0))
failures = int(root.get("failures", 0))
errors = int(root.get("errors", 0))
passed = total - failures - errors
pass_rate = f"{(passed/total*100):.1f}%" if total > 0 else "0%"

# 提取失败用例
failed_cases = []
for testcase in root.findall(".//testcase"):
    failure = testcase.find("failure")
    if failure is not None:
        failed_cases.append({
            "name": testcase.get("name"),
            "classname": testcase.get("classname"),
            "reason": failure.text[:200]  # 截取前200字符
        })
```

#### 步骤 2：生成报告

```python
report_md = f"""# 测试报告

## 摘要

- **总数**：{total}
- **通过**：{passed}
- **失败**：{failures}
- **错误**：{errors}
- **通过率**：{pass_rate}

## 失败用例

"""
for case in failed_cases:
    report_md += f"### {case['name']}\n\n"
    report_md += f"**原因**：{case['reason']}\n\n"

report_md += f"""
## 结论与建议

"""
if failures == 0:
    report_md += "✅ 所有测试通过，建议合并代码。\n"
else:
    report_md += f"⚠️ 有 {failures} 个用例失败，建议人工介入检查。\n"

# 保存报告
with open("report/agent_summary.md", "w", encoding="utf-8") as f:
    f.write(report_md)
```

### 5.4 输出

**Markdown 报告**：`report/agent_summary.md`

**JSON 输出**（可选）：

```json
{
  "summary": {
    "total": 50,
    "passed": 48,
    "failed": 2,
    "pass_rate": "96%"
  },
  "failed_cases": [
    {
      "name": "用例名",
      "reason": "断言失败..."
    }
  ],
  "conclusion": "建议人工介入检查失败用例",
  "report_path": "report/agent_summary.md"
}
```

## 六、CI/Workflow 编排（GitHub Actions 示例）

### 6.1 完整 Workflow

```yaml
name: Multi-Agent Testing Pipeline

on:
  push:
    branches: [main]

jobs:
  # 1. 提交监听
  collect-changes:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run commit scope mapping
        run: |
          python scripts/commit_scope_mapping.py \
            --changed-list changed_list.txt \
            --mapping testcase/vega/_config/path_scope_mapping.yaml \
            --repo ${{ github.repository }} \
            --sha ${{ github.sha }} \
            --message "${{ github.event.head_commit.message }}" \
            --output listener_output.json
      - name: Upload listener output
        uses: actions/upload-artifact@v4
        with:
          name: listener-output
          path: listener_output.json

  # 2. 智能体 A（用例维护）
  agent-a:
    needs: collect-changes
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Download listener output
        uses: actions/download-artifact@v4
        with:
          name: listener-output
      - name: Run Agent A
        env:
          LLM_API_KEY: ${{ secrets.LLM_API_KEY }}
        run: |
          python agents/agent_a_case_maintenance.py \
            --input listener_output.json \
            --output agent_a_output.json
      - name: Upload Agent A output
        uses: actions/upload-artifact@v4
        with:
          name: agent-a-output
          path: agent_a_output.json

  # 3. 智能体 B（维度分析）
  agent-b:
    needs: collect-changes
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Download listener output
        uses: actions/download-artifact@v4
        with:
          name: listener-output
      - name: Run Agent B
        env:
          LLM_API_KEY: ${{ secrets.LLM_API_KEY }}
        run: |
          python agents/agent_b_dimension_analyzer.py \
            --input listener_output.json \
            --output agent_b_output.json
      - name: Upload Agent B output
        uses: actions/upload-artifact@v4
        with:
          name: agent-b-output
          path: agent_b_output.json

  # 4. 智能体 C（用例筛选）
  agent-c:
    needs: [agent-a, agent-b]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Download artifacts
        uses: actions/download-artifact@v4
        with:
          name: listener-output
      - uses: actions/download-artifact@v4
        with:
          name: agent-b-output
      - name: Run Agent C
        env:
          LLM_API_KEY: ${{ secrets.LLM_API_KEY }}
        run: |
          python agents/agent_c_case_selector.py \
            --listener-input listener_output.json \
            --dimensions-input agent_b_output.json \
            --output agent_c_output.json
      - name: Upload Agent C output
        uses: actions/upload-artifact@v4
        with:
          name: agent-c-output
          path: agent_c_output.json

  # 5. 执行测试
  run-tests:
    needs: agent-c
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: '3.x'
      - name: Install dependencies
        run: pip install -r requirement.txt
      - name: Download Agent C output
        uses: actions/download-artifact@v4
        with:
          name: agent-c-output
      - name: Run tests by batches
        run: |
          python agents/run_tests_by_batches.py \
            --input agent_c_output.json
      - name: Upload test results
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: report/

  # 6. 智能体 D（报告生成）
  agent-d:
    needs: run-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Download test results
        uses: actions/download-artifact@v4
        with:
          name: test-results
      - name: Run Agent D
        run: |
          python agents/agent_d_report_generator.py \
            --junit report/junit_report.xml \
            --allure-xml report/xml \
            --output report/agent_summary.md
      - name: Upload final report
        uses: actions/upload-artifact@v4
        with:
          name: final-report
          path: report/agent_summary.md
```

### 6.2 关键点

- **并行执行**：`agent-a` 和 `agent-b` 都依赖 `collect-changes`，可并行
- **依赖关系**：`agent-c` 需要等待 `agent-a` 和 `agent-b` 完成
- **降级策略**：任一智能体失败时，按文档 4.5 节策略降级

## 八、开发建议

### 8.1 开发顺序

1. **先实现智能体 B**（最简单，只有输入输出，无文件操作）
2. **再实现智能体 C**（依赖 B 的输出，需要调用 `get_cases`）
3. **然后实现智能体 A**（最复杂，需要文件读写和 Git 操作）
4. **最后实现智能体 D**（需要解析测试结果）
5. **最后集成到 CI/Workflow**

### 8.2 测试建议

- 每个智能体独立测试（mock 输入）
- 使用真实的 `listener_output.json` 样例测试
- 在本地先验证，再集成到 CI

### 8.3 调试建议

- 每个智能体输出详细的日志
- 保存中间结果（JSON）便于调试
- 使用 GitHub Actions 的日志查看执行过程

## 十、常见问题

### Q1: 智能体 A 如何知道要修改哪个 suite 文件？

A: 通过 `listener_output.json` 中的 `suggested_suites` 字段，对应 `suites/*.yaml` 文件的 `story` 字段。

### Q2: 如何处理用例依赖链？

A: 智能体 C 在输出 `case_names` 时，必须递归展开所有 `prev_case` 依赖，确保前置用例也被执行。

### Q3: 智能体失败时如何处理？

A: 按文档 4.5 节降级策略：
- B 失败 → 默认 `dimensions=["functional"]`
- C 失败 → 按 `scope+tags` 兜底筛选
- A 失败 → 跳过用例维护，继续执行

### Q4: 如何测试智能体？

A: 使用真实的 `listener_output.json` 样例，在本地运行每个智能体脚本，验证输出格式。
